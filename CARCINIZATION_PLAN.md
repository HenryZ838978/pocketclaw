# Project Carcinization (CrabAgent)

> **"甲壳类动物的进化只有一个终态，Agent 的演化也一样。"**

**传播标签：【Agent 演化的蟹化终态】**

---

## 〇、一句话定位

所有 Claw（OpenClaw、ZeroClaw、NanoClaw、IronClaw）都在给**坐在电脑前的开发者**做 Agent。

但世界上随身带电脑的人可能不超过 10%。剩下 90% 的人——你的客户、你妈、你的同事——他们的全部数字生活在**手机**里。

CrabAgent 是第一个为手机而生的 AI Agent 框架。

不是"桌面 Agent 的移动端适配"。是**从商业市场反推架构**：谁是用户 → 用户在哪里 → 用户用什么设备 → 架构服务于这个设备。

```
阶级幻象费理论推导链：

卖什么感觉？ → "我有一个随身管家"
谁买这个感觉？ → 普通人（不是开发者）
普通人的设备？ → 手机（68亿活跃 vs PC 20亿）
手机上能跑什么？ → 不能跑 Docker/Node.js/终端
所以架构必须？ → 极轻量、跨平台、WASM、事件驱动、云端+本地混合

结论：移动端不是 Phase 5 的"未来扩展"。移动端是 Day 1 的架构约束。
```

## 〇.一、名字与隐喻体系

- **项目代号:** Project Carcinization (蟹化计划)
- **产品名称:** **CrabAgent**
- **核心隐喻：**
  - **龙虾 (OpenClaw):** 原始、臃肿。拖着巨大的尾巴（全量 Context），毫无防御力（RCE 漏洞满天飞），而且**它被钉在厨房台面上**——离开那台电脑它什么都不是。
  - **螃蟹 (CrabAgent):** 进化终态。极其精简的流线体（动态按需上下文），坚硬的甲壳（WASM 沙箱），致命的双螯（精准路由），而且**它在海滩上横行**——它跟着你走，不是你走去找它。

## 一、为什么所有 Claw 都困在桌面上

### 1.1 Claw 生态现状（2026-03-10 调研）

| 框架 | 语言 | 绑定 | RAM | 冷启动 | 用户画像 |
|------|------|------|-----|--------|---------|
| OpenClaw (294K★) | TypeScript | Node.js + Docker | 1.2GB | ~8s | 极客、开发者 |
| ZeroClaw (25K★) | Rust | 原生二进制 + 终端 | <5MB | <10ms | Rust 硬核开发者 |
| NanoClaw (20K★) | TypeScript | Docker/Apple Container | ~150MB | ~3s | Claude 用户 |
| IronClaw (8.7K★) | Rust | NEAR AI Cloud TEE | ~30MB | ~2s | 企业 CTO |
| FastClaw (141★) | Go | Kubernetes | - | - | DevOps 团队 |

**共同点：每一个都假设用户坐在一台有终端、有 Docker、有 root 权限的电脑前。**

这不是技术限制。这是创始人偏见——他们全是开发者，给自己做工具。

### 1.2 数字现实

- 全球活跃智能手机：**~68 亿**
- 全球活跃 PC：**~20 亿**
- 随身携带笔记本电脑的人口占比：**< 10%**
- 每天打开手机的次数：**平均 96 次**
- 每天打开笔记本电脑的次数：**平均 1-2 次**

"我有一个 AI 管家"这个感觉，如果管家被锁在你家书房的电脑里、你出门就联系不上它——这不是管家，这是台式机上的一个后台进程。

**管家的定义是：你在哪，它在哪。** 手机是唯一满足这个条件的设备。

### 1.3 OpenClaw 的安全灾难反而加强了移动端论点

- CVE-2026-25253：一键 RCE，42,000 实例被打穿，135,000+ 暴露在公网
- ClawHub 36% 的 Skill 带恶意载荷，1,184 个投毒包
- 512 个漏洞（2026-01 审计），含沙箱逃逸、命令注入、SSRF

这些漏洞的根源是什么？**Agent 跑在一台拥有完整文件系统、shell 权限、网络权限的通用计算机上。** 攻击面太大了。

反过来看：手机的 App 沙箱是 OS 级强制的。iOS/Android 每个 App 天生隔离，无法访问其他 App 数据，无法执行系统命令。**手机的安全模型天然就是 CrabAgent 想要的"甲壳"。** 你不需要自己造沙箱——操作系统已经帮你造好了。

## 二、钢剑：怎么卖 (The Illusion Fee)

### 2.1 所有 Claw 卖的是什么感觉

| 框架 | 卖的感觉 | 问题 |
|------|---------|------|
| OpenClaw | "我有一个 AI 助手" | 信号已反转——"我是那个被黑的人" |
| NanoClaw | "我知道不安全所以换了" | 逃跑信号，不是优越信号。锁死 Anthropic |
| ZeroClaw | "我用 Rust 我很硬核" | 圈外没人懂，没有视觉信号出口 |
| IronClaw | "我在企业级安全环境" | CTO 的故事，不是普通人的故事 |

共同问题：**它们全在卖给开发者。** 开发者是多大的市场？全球约 2800 万。智能手机用户是多大的市场？**68 亿。** 差了 243 倍。

### 2.2 CrabAgent 要卖什么感觉

**"我口袋里有一个管家。"**

不是"我有一个跑在我 Mac 上的 AI Agent"。是"我拿出手机，它已经帮我把事情办好了"。

这个感觉的强度跟设备的**贴身程度**成正比：
- 台式机上的 Agent → "我有一个会干活的软件"（弱，跟 Notion 差不多）
- 笔记本上的 Agent → "我有一个助手"（中等，但你合上盖子它就消失了）
- **手机上的 Agent → "我有一个随身管家"**（强，它永远在你身边，你睡觉它值夜班）

这就是为什么阶级幻象费在移动端是天花板级别的——**你从来不会把管家锁在书房里，你走到哪管家跟到哪。**

### 2.3 三层设计

**① 功能锚点要硬**
实打实的能力——帮你管消息、管日程、管邮件、管提醒。不是桌面式的 `execute_shell`，是手机原生场景：
- "帮我把今天的会议推后一小时，告诉对方我在路上"
- "这个群消息太多了，帮我总结一下重点"
- "我刚拍的这张收据，帮我记到账本里"

不需要炫酷。需要**真的省事**。

**② 信号出口要可截图**
移动端的信号出口比桌面强 100 倍——因为手机截图→发朋友圈/微博/推特是一个0摩擦动作：
- 通知栏弹出：`🦀 CrabAgent 已完成：为您今日处理 12 条消息，节省约 45 分钟`
- 月度报告卡片（精美设计，适合分享）：本月管家为你处理了 347 件事
- Widget 桌面小组件：实时显示管家状态

每一个都是天然的社交素材。这不是 CLI Dashboard 的极客截图——这是**普通人能看懂、想转发的东西**。

**③ 定价：从"开发者开源工具"到"订阅制个人服务"**
- **免费版**：基础能力，每天 20 次交互。足够让人体会"管家感"。
- **Pro 版**（$4.9/月）：无限交互 + 多平台同步 + 高级技能 + 定制人设。注意这里的锚定心理不再是"省 Token 钱"（普通人不知道 Token 是什么），而是"每月 35 块钱请了一个随身管家，太值了"。
- **价格参照物不是 OpenClaw 的 API 成本，是请一个真人助理的价格。** 真人助理月薪 3000+。CrabAgent $4.9/月。阶级幻象费率在这个对比下是合理的。

## 三、银剑：技术架构 (Technical Architecture)

架构由商业约束反推。商业约束是：**必须在手机上跑，68 亿人能用，极低资源占用。**

### 3.1 整体架构：云脑 + 端侧甲壳

```
┌─────────────────────────────────────────────────────────┐
│                    Cloud Brain (云脑)                     │
│                                                          │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ Semantic  │  │   Context    │  │    LLM Gateway    │  │
│  │ Router    │  │   Fetcher    │  │ (multi-provider)  │  │
│  │ (左螯)    │  │   (右螯)     │  │                   │  │
│  └────┬─────┘  └──────┬───────┘  └────────┬──────────┘  │
│       └───────────┬────┘                   │             │
│              ┌────┴────┐                   │             │
│              │ Reasoner │←──────────────────┘             │
│              │ Kernel   │                                │
│              └────┬─────┘                                │
│                   │ JSON-RPC                             │
│              ┌────┴─────┐                                │
│              │ WASM     │                                │
│              │ Sandbox  │  ← Skill 在这里跑              │
│              └──────────┘                                │
└─────────────────────┬───────────────────────────────────┘
                      │ HTTPS / WebSocket
                      │ (轻量协议，省电省流量)
┌─────────────────────┴───────────────────────────────────┐
│               Mobile Shell (端侧甲壳)                     │
│                                                          │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ 通知监听  │  │  本地缓存     │  │   UI 层           │  │
│  │ 事件采集  │  │  离线队列     │  │ (通知/Widget/App) │  │
│  └──────────┘  └──────────────┘  └───────────────────┘  │
│                                                          │
│  OS 级沙箱已由 iOS/Android 提供 — 无需自建甲壳           │
└─────────────────────────────────────────────────────────┘
```

**为什么是这个架构：**

| 约束 | 推导 |
|------|------|
| 手机不能跑 Docker / Node.js daemon | → 重计算放云端 |
| 手机杀后台进程 | → 端侧只做事件采集+缓存+展示，不做推理 |
| 手机 OS 已有沙箱 | → 甲壳不需要在端侧自建（iOS/Android 天然隔离） |
| 手机网络不稳定 | → 离线队列 + 断线重连 + 本地缓存 |
| 省电省流量 | → 双螯精准夹取（上下文极小化）从省钱变成省电的刚需 |
| 68 亿用户不会用终端 | → UI 是原生通知 + Widget + 极简 App，不是 CLI |

### 3.2 甲壳防御 (The Carapace)

**桌面版的问题：** OpenClaw 在一个不安全的 Node 进程里裸跑所有 Skill，所以需要造沙箱。

**移动端的真相：** iOS/Android 的 App 沙箱已经是 OS 级强制隔离。每个 App 不能访问其他 App 数据、不能执行系统命令、不能未经授权访问文件系统。**手机操作系统本身就是你的甲壳。**

CrabAgent 的甲壳设计因此变成两层：
1. **端侧：** 借用 OS 沙箱。CrabAgent 作为一个普通 App，天然被隔离。不造轮子。
2. **云侧：** Skill 在 WASM Sandbox（Wasmtime）中执行。推理内核与执行沙箱分离（双内核）。这是因为云端没有 OS 级沙箱帮你，所以自己造。

### 3.3 双螯夹取 (The Pincers)

跟桌面版一样的原理，但移动端让它从"优化"变成了"刚需"：

- **左螯 (Semantic Router):** <50ms 意图分类。省 Token = 省钱，在桌面上是优化；在移动端 = **省电 + 省流量**，直接影响用户体验。
- **右螯 (Context Fetcher):** 只拼装最相关的 1-3 个 Skill 上下文。移动端网络传输量直接减少 90%。

在 4G/5G 环境下，全量 Context 传上去再传下来的延迟和流量消耗，对手机用户是不可接受的。双螯在移动端不是锦上添花——是 table stakes。

### 3.4 八足横行 (Event-Driven)

手机天生就是事件驱动的——推送通知、Intent、Shortcut、后台 fetch 全是事件机制。

CrabAgent 端侧 App 的核心循环：
1. **监听事件**：新消息通知、日历提醒、位置变化、用户主动唤起
2. **采集上下文**：把事件+必要信息打包成极精简的 JSON
3. **送云脑处理**：HTTPS 单次请求，云脑用双螯路由+推理+沙箱执行
4. **接收结果**：云脑返回行动指令（回复消息/设置提醒/展示信息）
5. **本地执行**：调用系统 API 完成动作（发通知/更新 Widget/回复）

不需要 Heartbeat 长轮询。推送通知就是最好的事件驱动机制——Apple/Google 已经帮你建好了。

### 3.5 技术栈选型

| 层 | 选择 | 理由 |
|---|---|---|
| 云端核心 | **Rust** | 极低延迟、内存安全、WASM 原生支持。Ferris 是螃蟹（meme 对齐）|
| 云端沙箱 | **Wasmtime** | 亚毫秒启动、~2MB/实例、能力基权限 |
| 云端 Embedding | **text-embedding-3-small / nomic-embed** | 语义路由核心 |
| 端侧 Android | **Kotlin** | Android 原生，Notification Listener / Accessibility / Widget |
| 端侧 iOS | **Swift** | iOS 原生，UNNotification / WidgetKit / Shortcuts |
| 端侧跨平台 UI | **Kotlin Multiplatform** or **Flutter**（备选） | 如果不想维护两套 |
| 通信协议 | **HTTPS + Server-Sent Events** | 比 WebSocket 省电，兼容推送 |
| 离线缓存 | **SQLite** | 两端通用 |

## 四、竞争格局：为什么这个空位存在

### 为什么没人做手机端 Agent

1. **创始人偏见。** Claw 生态的创始人全是开发者。开发者的本能是"帮我在终端里自动化"。他们想不到普通人的需求。
2. **技术路径依赖。** OpenClaw 的 430K LoC TypeScript + Docker 架构根本无法移植到移动端。ZeroClaw 的 native Rust 二进制绑定 OS API。NanoClaw 绑定 Docker。**它们要上移动端都得重写。**
3. **"工具"心态 vs "服务"心态。** 所有 Claw 卖的是"一个开源工具"。CrabAgent 卖的是"一个订阅服务"。工具默认在开发者的电脑上跑。服务默认在用户的口袋里跑。

### CrabAgent 的结构性优势

| 维度 | 桌面 Claw 们 | CrabAgent |
|------|-------------|-----------|
| 目标用户 | 开发者（2800 万） | 智能手机用户（68 亿） |
| 设备 | MacBook / Linux | 手机（随身、永远在线） |
| 安全模型 | 自建沙箱（WASM/Docker） | 端侧用 OS 沙箱 + 云侧 WASM |
| 上下文效率 | 省钱（nice to have） | 省电省流量（must have） |
| 事件驱动 | 自建 Webhook | OS 推送通知（已有基础设施） |
| 阶级幻象感 | "我有一个跑在 Mac 上的后台进程" | **"我口袋里有一个管家"** |
| 信号出口 | CLI 截图（给极客看） | 通知栏 + Widget + 月报卡片（给所有人看） |

## 五、可行性与路线图

### 靠谱程度分析：★★★★★

1. **顺应必然趋势：** Agent 从开发者工具走向大众消费品是必然路径，跟 PC→手机、命令行→GUI 是同一条进化线。
2. **竞争空位真实存在：** 68 亿手机用户 vs 0 个移动端原生 Agent 框架。这不是细分市场，是主战场被所有人忽略了。
3. **架构天然适配：** Rust + WASM + 事件驱动 + 极低内存——这套组合在桌面上是"比别人好"，在移动端是"只有这样才能跑"。
4. **阶级幻象费天花板更高：** "口袋里的管家" >> "Mac 上的后台进程"。普通人愿意为前者付费，不愿意为后者付费（因为后者他根本不知道是什么）。

### MVP 路线图：8-10 周

| Phase | 核心任务 | 周期 | 交付物 |
|---|---|---|---|
| **Phase 0: PoC** | Wasmtime 云端沙箱 PoC + Embedding 路由准确率验证 + 端云通信 PoC | 1 周 | 三个可跑的 PoC |
| **Phase 1: 云脑** | Rust 云端服务：双螯路由 + WASM 沙箱 + LLM 接口 + 3 个核心 Skill（消息总结、日程管理、提醒） | 3 周 | 可调用的云端 API |
| **Phase 2: 端壳** | Android App（Kotlin）：通知监听 + 事件采集 + 云脑通信 + 结果展示（通知+Widget） | 2 周 | 可用的 Android App |
| **Phase 3: 闭环** | 端到端全链路：手机收到消息 → 云脑处理 → 手机展示结果/执行动作 | 1-2 周 | 完整 Demo |
| **Phase 4: 包装** | 信号出口设计（月报卡片、Widget 美化）+ Landing Page + 内测发布 | 1 周 | 可对外展示的产品 |

**为什么先 Android 不先 iOS：**
- Android 的通知监听（NotificationListenerService）和辅助功能（AccessibilityService）比 iOS 开放得多
- 不需要 Apple 开发者账号和审核周期
- 全球 Android 市占率 ~72%
- APK 侧载分发，不依赖 App Store
- iOS 版作为 Phase 5 后续跟进

### 战略建议

**不要对标 OpenClaw 做 benchmark 了。** 那是开发者的语言。

对标**普通人每天的痛苦**：
- "群消息 999+ 我不想看但怕错过重要的" → CrabAgent 消息摘要
- "我老忘记回消息，结果别人以为我故意不回" → CrabAgent 自动提醒+草稿
- "收据、发票、截图堆了一手机，从来不整理" → CrabAgent 自动归档

Launch 姿势不是 GitHub README + benchmark 表格。是一个 30 秒的手机录屏：消息进来 → 通知栏弹出管家摘要 → 一键回复。**普通人看得懂的，才叫产品。**


Day 1-2：Rust workspace 搭建 + Wasmtime 沙箱 PoC（验证甲壳技术可行性）

Day 2-3：Embedding 语义路由 PoC（验证左螯准确率 ≥ 90%）

Day 3：端云通信 JSON 协议设计（手机和云脑之间说什么格式的话）

Day 4-5：Android 项目骨架搭建（Kotlin，NotificationListenerService，在模拟器上验证结构）

Day 5-6：LLM 多模型接口适配（Claude / GPT / DeepSeek 统一 trait）

Day 6-7：第一个 Skill message_triage（消息分诊）在命令行跑通全链路

目标是：手机到手那天，开箱 → 插线 → adb install → 授权 → 看到管家工作，30 分钟内。 因为所有云端代码和 Android 骨架已经就绪，手机只是把输入源从模拟 JSON 换成真实通知。
